# BLI Exercise Authoring Guide

This is the working manual for authoring a BLI exercise that meets the locked architectural contract. Read this once before you start your first exercise. Refer back as you draft.

If anything here conflicts with `BLI-ADR-001`, the ADR wins. If you find a conflict, flag it to Chris before publishing.

---

## 1. The architecture every exercise inherits

### 1.1 The bifurcation — what lives where

Every exercise lives across two surfaces. Knowing what goes where is the single most important thing in this guide.

| Surface | Role | What it does | What it never does |
| :-- | :-- | :-- | :-- |
| **Circle** (the embed) | Passive orchestrator | Step prose, copy-prompt blocks, "where to go next" card | No AI. No forms. No questionnaires. No artifact uploads. No paste-back collection. |
| **Learner's LLM** (Claude default) | Active worker | All generation, iteration, project creation, custom-instructions install (from author-supplied compiled block) | This is where the persistent takeaway lives after the exercise ends. |

If you ever feel the urge to add an input field, a quiz form, or a "submit your work" button to the Circle side — stop. That's a contract violation. The learner's LLM is the only place AI work happens.

### 1.2 The 3-layer architecture you are teaching

Every BLI exercise is a concrete instance of the same underlying mental model:

| Layer | What it is | How often it changes |
| :-- | :-- | :-- |
| **Stable System** | Custom Instructions (rules, tone, format, process) | Rarely |
| **Stable Context** | Source docs uploaded to the Project (facts to cite) | When life/facts change |
| **Volatile Input** | The per-run input (a JD, a topic, an idea) | Every run |

Your exercise must teach this pattern through the act of building one working instance of it. By the end, the learner has:

1. A named Claude Project (per-exercise; never shared across exercises) that survives after the exercise.
2. Custom Instructions installed in that Project at Step 2 (Stable System) — pre-compiled, not generated by Claude during the exercise.
3. Source docs uploaded if relevant (Stable Context).
4. A working flow (one or more prompts) the learner re-runs with new Volatile Input.

**v2.0 install timing.** Under v2.0 (locked 2026-05-07), the Custom Instructions are **installed up-front** at Step 1 or 2, not generated by the Coach at the end of the exercise. Reproducibility for cohorts of hundreds-to-thousands requires consistent personality baked into the CI, not re-asserted per prompt. See §1.4 for how the compiled CI is authored.

### 1.3 The benchmark approach

Every exercise is reverse-engineered from a working protocol. You do not freehand new prompts and hope they work. You start with a tested artifact and build the exercise to produce that artifact reliably.

For Exercise 01, the benchmark is the **CBM Resume Writer Protocol** — a real, working set of Custom Instructions Chris uses. The exercise teaches the learner to produce something equivalent on their own resume.

For Exercise 07 (Industry Researcher, the first v2.0-conforming reference), the benchmark is the structural toolkit + queryset specified in BLI-ADR-001 §Appendix B (NVP Track Coach) — applied to the learner's specific venture inputs.

Before you draft your exercise, name your benchmark. If you don't have one, build one first (use it on real material; iterate until it works), then write the exercise that teaches the learner to produce one for themselves.

### 1.4 CI authoring architecture (v2.0; refined v2.1)

Every per-exercise Custom Instructions block has three layers. **Two compilation patterns** are valid; choose per the exercise's teaching arc.

| Layer | Source |
| :-- | :-- |
| **BLI Coach Base** | ADR-001 Appendix A (verbatim) |
| **Track Coach** | ADR-001 Appendix B for NVP; future appendices for PM/Vibe |
| **Per-Exercise Coach Role** | Either pre-built by the author **or** student-generated via an encapsulation prompt |

**Pattern 1 — Pre-compiled.** Author concatenates all three layers; learner pastes the full block at Step 2. No encapsulation step. Use when the role embeds structured tools / frameworks / queries / deterministic logic the student wouldn't reliably derive (e.g., Ex07's NVP queryset).

**Pattern 2 — Base + student-encapsulated.** Author pre-builds BLI Base + Track Coach (if applicable); learner installs that at Step 2. Per-Exercise Coach Role is generated by the student through the exercise and **appended below the Base** in Project Custom Instructions at the end. Use when the teaching arc IS "build your own Protocol from your own preferences" (e.g., Ex01's Resume Writer; likely Ex09, Ex12).

**Selection rubric:** see ADR-001 §5.8.

**When components change:** If BLI Coach Base or a Track Coach is amended, every dependent exercise's CI block(s) must be regenerated and re-versioned. CIs name the versions of their parents.

### 1.5 Single-chat default (v2.0)

By default, every BLI exercise runs in a **single chat** inside its Project. Phases (interview → research → diagnosis → memo → drill-down) progress sequentially through copy-paste prompts the learner runs in that one chat.

Multi-chat structures (e.g., a Coach chat + a Research Scratchpad chat) are permitted only when the per-exercise spec documents a structural reason. Default is single-chat; multi-chat must be argued.

Existing pre-v2.0 references (Ex12 Sharpener + Scratchpad) used multi-chat without documented justification. They are slated for retrofit.

## 2. The Show + Tell + Give pattern

Every exercise has three required phases. Most also have a fourth optional phase: Extend.

| Phase | What happens | Where it happens | Goal |
| :-- | :-- | :-- | :-- |
| **Show** | Circle instructs → Claude produces → learner sees it work on their real material | Both surfaces | Proof, not theory. |
| **Tell** | Reveal prose names the architecture they just used | Circle | The learner now understands *why* it worked. |
| **Give** | Circle instructs → learner installs author-compiled Custom Instructions in a named Project | Both surfaces | The takeaway lives in their LLM after the exercise ends. |
| **Extend** *(optional)* | Apply the same pattern to a different domain via a generator prompt | Both surfaces | Pattern transfer to the learner's other repetitive tasks. |

**Phase order is flexible.** Ex01 runs Show → Tell → Give → Extend. Ex09 puts a short Tell early (Step 2). Ex12 distributes Tell across multiple steps.

**Under v2.0 the Give phase splits into two moments:** **Give (install)** happens at Step 1 or 2 — the learner pastes the pre-compiled CI block into Project Custom Instructions. **Give (reveal)** happens at the end — a brief section names what the learner now permanently has (the persistent Coach), how to re-run, and when it pays to come back. Ex07 demonstrates this two-moment pattern.

**All three required phases must be present** in some form. Pick whatever ordering serves the topic best.

### 2.1 What's never permitted as a phase

These are explicitly disallowed as exercise pattern elements (per ADR §4):

- Mid-flow Inspect stages
- Artifact uploads to Circle
- Paste-back collection on Circle
- Learner questionnaires on Circle

If your exercise feels like it needs one of these, the design is wrong. Re-route the work to the learner's LLM.

## 3. The file contract

Every exercise ships as **two paired files** at the repo root. Filename pattern: `exNN-<slug>` where `NN` is the zero-padded exercise number and `<slug>` is a short kebab-case identifier.

| File | Purpose |
| :-- | :-- |
| `exNN-<slug>.html` | The standalone interactive exercise page. Self-contained: inline CSS, vanilla JS, no external scripts, no CDN. |
| `exNN-<slug>.oembed.json` | The oEmbed metadata that tells Circle (via Iframely) how to render the page as a rich iframe. |

### 3.1 Required `<head>` tags

Every HTML file's `<head>` must include exactly these tags, with `NN` and `<slug>` replaced for your exercise. **All URLs must use `exercises.bigleverinstitute.org` — never `bigleverventures.github.io`.** Iframely will reject the embed if the provider URL doesn't match.

```html
<meta charset="UTF-8">
<meta name="viewport" content="width=device-width, initial-scale=1.0">
<title>Exercise NN — <Title> | Big Lever Institute</title>

<!-- Canonical URL -->
<link rel="canonical" href="https://exercises.bigleverinstitute.org/exNN-<slug>.html">

<!-- Open Graph -->
<meta name="description" content="<short description>">
<meta property="og:type" content="website">
<meta property="og:title" content="Exercise NN — <Title>">
<meta property="og:description" content="<short description>">
<meta property="og:url" content="https://exercises.bigleverinstitute.org/exNN-<slug>.html">
<meta property="og:site_name" content="Big Lever Institute">

<!-- Twitter Card -->
<meta name="twitter:card" content="summary_large_image">
<meta name="twitter:title" content="Exercise NN — <Title>">
<meta name="twitter:description" content="<short description>">

<!-- oEmbed discovery -->
<link rel="alternate" type="application/json+oembed" href="https://exercises.bigleverinstitute.org/exNN-<slug>.oembed.json" title="Exercise NN — <Title>">
```

### 3.2 oEmbed JSON template

Leave `height` as a placeholder (e.g., `0` or `"PLACEHOLDER"`). The admin's `npm run measure` script measures the actual rendered height and patches it in. Do not guess heights — they are always wrong.

```json
{
  "version": "1.0",
  "type": "rich",
  "provider_name": "Big Lever Institute",
  "provider_url": "https://exercises.bigleverinstitute.org/",
  "title": "Exercise NN — <Title>",
  "author_name": "Big Lever Institute",
  "author_url": "https://exercises.bigleverinstitute.org/",
  "width": 1080,
  "height": 0,
  "html": "<iframe src=\"https://exercises.bigleverinstitute.org/exNN-<slug>.html\" width=\"100%\" height=\"0\" frameborder=\"0\" allow=\"clipboard-write\" style=\"border:none;display:block;max-width:100%\" title=\"Exercise NN — <Title>\" loading=\"lazy\"></iframe>"
}
```

### 3.3 Footer format

Last element before `</body>`:

```html
<footer>
  <p>© Big Lever Institute · Exercise NN · vX.Y · YYYY-MM-DD · Governed by BLI-ADR-001</p>
</footer>
```

Use absolute date format (e.g., `2026-05-06`). Increment the version each time you ship a meaningful copy change.

## 4. The design system (locked)

These are not stylistic preferences — they are the inherited design contract. The skeleton template carries the complete CSS verbatim. Don't edit the `<style>` block unless you've cleared the change with Chris.

### 4.1 Color palette

| Token | Hex | Used for |
| :-- | :-- | :-- |
| `--cream` | `#F7F4EE` | Page background |
| `--surface` | `#FFFFFF` | Callouts, next-card |
| `--text` | `#2B2420` | Body text |
| `--muted` | `#6B6560` | Secondary text |
| `--border` | `#E8E0D4` | Soft borders |
| `--gold` | `#B8924F` | Deliverable accent |
| `--gold-bg` | `#F5EDD4` | Deliverable card background |
| `--gold-border` | `#D9BE85` | Deliverable card border |
| `--purple` | `#5D4E8B` | Prompt accent + copy button |
| `--purple-bg` | `#ECE8F2` | Prompt block background |
| `--purple-border` | `#C0B5D8` | Prompt block border |
| `--success` | `#3B7A3B` | "Copied" confirmation state |

### 4.2 Visual rules

- **Background: cream.** Never dark.
- **Layout: 760px max-width, centered, generous whitespace.** Responsive down to ~360px.
- **Typography: system stack** (`-apple-system, BlinkMacSystemFont, "Segoe UI", Roboto…`). No web fonts.
- **Prompt copy surfaces: purple.** Plain-prose intro before the block (not a callout box).
- **Deliverable moments (install checklists, worked examples): gold.**
- **Callouts (parenthetical guidance): white surface, muted left border.**
- **No multicolored text. No emoji decoration. No icon fonts.**

## 5. Step structure conventions

Every exercise is a sequence of `<section class="step">` elements. Each step has an eyebrow label and a heading.

### 5.1 Step header pattern

```html
<section class="step" id="step-N">
  <div class="step-header">
    <span class="step-num">Step N · <Phase Label></span>
  </div>
  <h2><Step heading — verb-led, concrete></h2>
  ...
</section>
```

Phase label examples (v2.0): `Setup`, `Install`, `Open chat`, `Show`, `Show (refine)`, `Reveal` (or `Tell`), `Give`, `Extend`. Step 0 is always "Before you start" — prereqs, time estimate, account requirements. (`Give (encapsulate)` is deprecated under v2.0; the CI is pre-compiled, not generated by Claude in-chat.)

### 5.2 Suggested step count

- **8–11 steps** for v2.0 exercises. The CI install (Step 2) and chat-setup (Step 3) account for two more numbered steps than pre-v2.0 exercises had. Ex07 has 11 (Step 0 through Step 10).
- Pre-v2.0 references (Ex01 has 9, Ex09 has 9, Ex12 has 9) used a tighter step count because they generated the CI at the end via an encapsulation prompt — that pattern is deprecated.
- One instruction per step. If a step balloons, split it.

### 5.3 What each phase contains (v2.0; refined v2.1)

Step ordering depends on the CI pattern (see §1.4 and ADR §5.8).

**Pattern 1 step content (e.g., Ex07):**

| Phase | Typical content |
| :-- | :-- |
| Step 0 — Before you start | What to gather, what account, time estimate |
| Step 1 — Setup | Create the per-exercise Project, name it |
| Step 2 — Install (Give) | Gold-card install checklist + purple `<pre>` block containing the **full compiled CI** (Base + Track + Coach Role) |
| Step 3 — Open chat | Click New Chat; enable any per-chat settings (e.g., web research) |
| Step 4+ — Show | Starter prompts; placeholders for volatile input |
| Show (refine) | Optional sharpening prompts |
| Tell | The layer visualization (`<div class="layer">`) |
| Give (reveal) | Gold card or prose: what the learner permanently has, how to re-run |
| Extend (optional) | Worked example + generator prompt |
| Final step — Where to go next | next-card |

**Pattern 2 step content (e.g., Ex01):**

| Phase | Typical content |
| :-- | :-- |
| Step 0 — Before you start | What to gather, what account, time estimate |
| Step 1 — Setup | Create the per-exercise Project, upload source docs |
| Step 2 — Install Base (Give, partial) | Gold-card install checklist + purple `<pre>` block containing **BLI Base + Track Coach (if applicable) only** |
| Step 3 — Open chat | Click New Chat |
| Step 4+ — Show | Starter prompt(s); the student does the actual work |
| Show (refine) | Optional sharpening prompts |
| Give (encapsulate) | The encapsulation prompt — student asks Claude to draft their per-exercise Protocol |
| Give (install — append) | Gold card: append the generated Protocol below the BLI Base in Project Custom Instructions; invoking prompt for future runs |
| Tell | The layer visualization — name the BLI Base + student's own Protocol as the two parts of Stable System |
| Extend (optional) | Worked example + generator prompt |
| Final step — Where to go next | next-card |

## 6. Copy-prompt conventions

The purple prompt block is the most critical UI element. It carries the prompts the learner pastes into Claude.

### 6.1 The prompt block structure

```html
<div class="prompt">
  <div class="prompt-label">Starter prompt — copy, paste, fill in the JD, send</div>
  <pre id="pN">[The full prompt text the learner will paste]</pre>
  <button class="copy-btn" data-copy="pN" type="button">Copy</button>
</div>
```

- `id` on the `<pre>` is unique per block (e.g., `p2`, `p3a`, `p4`).
- The copy button's `data-copy` attribute matches the `<pre>` id.
- The label is verb-led and tells the learner what to do.

### 6.2 Prompt copy rules (these are non-negotiable)

- **You author every prompt the learner will paste. No "ad-lib" fields.** If the learner needs to insert something (a JD, a topic, an idea), you mark it explicitly: `[PASTE THE JD HERE]`, `[DESCRIBE YOUR IDEA HERE]`.
- **Each prompt is paste-ready.** No instructions inside the `<pre>` block beyond what Claude needs.
- **No code fences inside `<pre>`.** The block is plain text.
- **Prompts must scaffold against your benchmark protocol, not freehand Claude output.** This is the §7 mitigation in the ADR.

### 6.3 Plain-prose intro before each prompt

Don't put the prompt in a callout box. Write a short paragraph of plain prose above the prompt that frames what's about to happen. Then the purple block sits in normal page flow.

Good (matches ex01 Step 2):

> In the new chat inside your Resume Writer Project, paste the starter prompt below, then paste your target JD where indicated. Claude will use the resume from the Project's files as the source of your career history.
>
> [purple prompt block]

Bad (callout-wrapped intro):

> ⚠️ Important: paste this prompt and...

## 7. The Tell phase — visualizing the architecture

The Tell section uses `<div class="layer">` blocks with a left gold border. Three layers — Stable System, Stable Context, Volatile Input — each named, each with a one-paragraph explanation tied to the exercise's domain.

```html
<section class="step reveal" id="step-N">
  <div class="step-header"><span class="step-num">Step N · Tell</span></div>
  <h2>How this actually works</h2>
  <p>What you just built follows a simple architecture. It works for any task where you repeat similar work with varying inputs.</p>

  <div class="layer">
    <strong>Layer 1 — Stable System: <your-system-name>.</strong>
    [One paragraph relating it to this exercise's domain]
  </div>
  <div class="layer">
    <strong>Layer 2 — Stable Context: <your-context-name>.</strong>
    [One paragraph]
  </div>
  <div class="layer">
    <strong>Layer 3 — Volatile Input: <your-input-name>.</strong>
    [One paragraph]
  </div>
</section>
```

The Tell prose names the transferable pattern in the learner's vocabulary, not abstract jargon.

## 8. The Give phase — install + reveal (Pattern 1) or install Base + encapsulate + append (Pattern 2)

Under v2.0/v2.1, Give shape depends on CI pattern (§1.4).

### 8.1 Pattern 1 — Give (install) at Step 2 — gold card + compiled CI block

The install moment sits in a gold card adjacent to a purple `<pre>` block containing the full compiled CI.

```html
<div class="gold-card">
  <div class="gold-label">Deliverable — your permanent <Project Name> Coach</div>
  <ol>
    <li>Inside your <strong><Project Name></strong> Project, open <strong>Project settings</strong>.</li>
    <li>Open <strong>Custom Instructions</strong>.</li>
    <li>Copy the entire block below. Paste. <strong>Save.</strong></li>
  </ol>
</div>

<div class="prompt">
  <div class="prompt-label">Custom Instructions — copy, paste into Project settings, save</div>
  <pre id="ci">[FULL COMPILED CI: BLI Coach Base + Track Coach + Per-Exercise Coach Role, concatenated]</pre>
  <button class="copy-btn" data-copy="ci" type="button">Copy</button>
</div>
```

The compiled CI is the source-of-truth artifact. The author maintains the three layers separately (per §1.4) but ships one block.

### 8.2 Give (reveal) at end — the persistent capability

After Show and Tell, a final gold card (or short prose section) names what the learner now permanently has and how to re-run. The CI was already installed; this section makes the persistence explicit.

```html
<div class="gold-card">
  <div class="gold-label">How to re-run with new inputs</div>
  <ol>
    <li>Open your <strong><Project Name></strong> Project.</li>
    <li>Click <strong>New Chat</strong>.</li>
    <li>(any per-chat settings needed)</li>
    <li>Paste the starter prompt again with updated inputs.</li>
    <li>...</li>
  </ol>
</div>
```

Plus a "use it whenever..." list of triggering moments (region change, price change, competitor moves, regulatory shifts, etc.). Ex07 §Step 9 is the canonical example.

### 8.3 Pattern 2 — Give (encapsulate) + Give (install/append)

For Pattern 2 exercises (Ex01-style; teaching arc = "build your own Protocol"), Give happens in two moments at the end of the exercise:

- **Give (encapsulate):** an encapsulation prompt asks Claude to draft the per-exercise Protocol (Custom Instructions content) based on what worked during the Show steps.
- **Give (install/append):** a gold-card install checklist directs the learner to **append** the generated Protocol below the existing BLI Base in their Project Custom Instructions. (Don't replace — append.)

Pattern 2 retains the encapsulation prompt because the teaching arc IS the act of building one's own Protocol. The encapsulation prompt should produce content that **complements**, not duplicates, the BLI Base operating principles already installed at Step 2.

### 8.4 Invoking prompt

Both patterns may include a short, repeatable invoking prompt for future runs (e.g., Ex01's "Using my Resume Writer Protocol, generate a tailored resume and cover letter for the JD below"). Optional — many exercises run their phase logic via the same starter prompts re-pasted.

### 8.5 What's gone from v1.x

What's gone is "no Custom Instructions installed until the end." Both v2.0 patterns install BLI Base at Step 2 up-front, so cohort reproducibility is consistent from the first prompt forward. Pattern 1 installs everything up-front; Pattern 2 installs Base up-front and appends the Protocol later.

## 9. The Where-to-go-next card

The final step is always a `<div class="next-card">`. It contains short pointers to:

- A deeper Master Class on the same architecture (typically MC6 — Prompt Engineering Mastery).
- Adjacent exercises (placeholder `[TBD]` is acceptable until the cross-references are built).
- An optional one-paragraph reminder that frames the takeaway.

No numerical "next/previous" navigation chrome. No progress bar. **Don't add a sticky step-progress bar at the top of the page** — this is a hard rule and was explicitly stripped from ex09.

## 10. The authoring workflow inside Cowork

A suggested phasing for your authoring conversations inside the Project:

| Phase | Goal | Output |
| :-- | :-- | :-- |
| **A. Articulate** | Name the topic, the benchmark, the Project name, the learner's takeaway | One-paragraph framing + the PMF formulation if relevant |
| **B. Build the benchmark** | Use Claude on real material to produce a working version of what you'll teach | The actual artifact (memo, study notes, PMF statement, etc.) |
| **C. Compile the CI** | Concatenate BLI Coach Base (ADR Appendix A) + Track Coach (ADR Appendix B for NVP) + Per-Exercise Coach Role into one paste-ready block | The compiled CI, versioned (e.g., Ex07 CI v1.0) |
| **D. Spec** | Lock copy for every per-chat prompt the learner will paste | A spec doc with all locked copy |
| **E. Codify (HTML)** | Copy `template-exNN-skeleton.html` and fill it in step-by-step, embedding the compiled CI in the Step 2 block | The two paired files |
| **F. Self-check** | Run `02-Pre-Handoff-Checklist.md` | A clean handoff or a list of fixes |
| **G. Hand off** | Deliver the two files to Chris | Chris publishes |

You can do A through G all inside one Cowork Project. Each phase is a conversation in the same Project. The Project's file library accumulates the spec doc, the compiled CI, and any reference material.

## 11. Common pitfalls

| Pitfall | Why it breaks | Fix |
| :-- | :-- | :-- |
| Adding a form/input on the Circle side | Violates bifurcation; Circle has no AI | Re-route to learner's LLM via a copy-prompt |
| Using a CDN, web font, or external script | Violates §5.5; breaks privacy posture | Inline CSS + vanilla JS only |
| Putting a sticky step-progress bar at the top | Visual chrome that adds no learning value | Remove it. Use eyebrow labels per step instead. |
| Guessing the iframe height | The admin's measure script always overrides; wrong guesses ship and clip content | Leave height as `0` or `PLACEHOLDER` |
| Using `bigleverventures.github.io` URLs | Iframely rejects mismatched provider domains | Always `exercises.bigleverinstitute.org` |
| Letting the learner ad-lib a prompt | Quality variance; defeats the benchmark approach | Author every prompt; mark insertion points with explicit placeholders |
| Mid-flow questionnaires or paste-back fields | Disallowed pattern element per ADR §4 | Convert to a copy-prompt or remove |
| Forgetting the footer date stamp | Loses the version-tracking signal | Always include `vX.Y · YYYY-MM-DD` |
| Sharing one Project across multiple exercises | One CI per Project; specialized Coach roles conflict | Each exercise gets its own dedicated Project per ADR §5.2 |
| Choosing Pattern 1 when the teaching arc is "build your own" — or Pattern 2 when the role embeds deterministic tooling | Wrong pattern for the exercise's job — students miss the lesson, or the Coach has no tools | Apply the §1.4 selection rubric. Pattern 2 keeps the encapsulation prompt; Pattern 1 doesn't. |
| Forgetting to install BLI Base at Step 2 | Loses cohort reproducibility from the first prompt onward | Both patterns install BLI Base up-front at Step 2; Pattern 2 just adds the Protocol later via append |
| Multi-chat structure without documented justification | Default is single-chat per ADR §5.9 | Justify in spec or convert to single-chat with phase markers |
| Re-asserting Coach role in every per-chat prompt | Wastes tokens; defeats the purpose of CI | Bake the role into the compiled CI; per-chat prompts focus on phase logic |

## 12. When to escalate

Ping Chris (don't guess) for any of these:

- A change to the locked CSS or design system
- A new pattern element not described in ADR §4
- An exercise topic where the benchmark approach doesn't seem to apply
- A use case where the bifurcation seems to require an exception
- An ADR amendment

Everything else — keep moving. The point of this kit is to let you author without bottlenecking on Chris for routine decisions.
